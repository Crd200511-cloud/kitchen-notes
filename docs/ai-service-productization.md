# AI 能力给别人使用的方案

## 目标

把当前“AI 食谱整理”能力从你自己的网站里抽出来，变成一个可复用的服务。

适用对象：

- 你自己的网站
- 你未来的微信小程序
- 别人的网站或小程序

## 不推荐的做法

- 不要把 OpenAI API key 放到前端
- 不要把 AI 能力写死在某一个网页里
- 不要先做插件再想后端

## 推荐形态

先做统一 AI 服务，再做多端接入。

### 服务层

统一接口：

- `POST /api/parse-recipe`
- 输入：原文、来源链接、导入类型
- 输出：结构化食谱 JSON

### 接入层

你自己的产品直接接这个接口。

- 网站接入
- 微信小程序接入
- 后续给别人开放 API 或 SDK

## 推荐的三层形态

### 第 1 层：内部接口

当前先服务你自己的产品。

### 第 2 层：SDK

给别人一个简单封装。

例子：

- Web SDK
- 小程序 SDK

### 第 3 层：插件

当真的出现合作方需求时，再做插件壳。

插件只是接入入口，不是 AI 本体。

## 对外 API 设计建议

### 接口

`POST /api/parse-recipe`

### 请求体

```json
{
  "rawText": "原始分享文案",
  "sourceUrl": "https://...",
  "mode": "note"
}
```

### 返回体

```json
{
  "title": "番茄炒蛋",
  "sourceUrl": "https://...",
  "ingredients": ["番茄 2个", "鸡蛋 3个"],
  "steps": [
    {
      "title": "准备食材",
      "detail": "番茄切块，鸡蛋打散备用",
      "duration": "",
      "tip": ""
    }
  ],
  "notes": "",
  "unresolved": []
}
```

## 给别人用时必须有的能力

### 1. 调用身份

至少支持以下一种：

- API Key
- App ID + Secret
- 用户登录态

### 2. 额度控制

必须控制：

- 每分钟调用次数
- 每日调用次数
- 每个应用的额度

### 3. 错误日志

至少记录：

- 请求时间
- 请求方
- 原文长度
- 模型名称
- 成功/失败
- 错误原因

### 4. 版本控制

建议保留模型版本和提示词版本。

## 推荐的产品化路径

### 阶段一：内部能力

- 只供你自己网站和小程序使用
- 统一后端接口
- 加基础日志

### 阶段二：伙伴接入

- 开放 API Key
- 提供文档
- 提供示例请求

### 阶段三：插件/SDK

- 小程序插件
- Web SDK
- 管理后台

## 小程序插件的现实定位

如果以后要做“AI 插件”，建议理解为：

- 插件负责 UI 和接入体验
- 统一 AI 服务负责真正的解析

这样你就不会被某个平台绑定死。

## 对你当前项目的建议

下一步不要先做插件。

正确顺序是：

1. 完成统一 AI 服务
2. 完成你自己的小程序前端
3. 等内部产品跑顺了，再抽 SDK 或插件
